# 房间机制与多人联机架构设计

## 1. 背景与目标

当前项目默认运行在“单服务器 = 单世界 = 单局游戏”的模型下：

- 所有玩家共用同一个 SQLite 数据库
- 市场、新闻、订单、账户等状态默认全局共享
- WebSocket 事件广播默认全局频道
- 玩家身份主要通过 `player_id` 推导
- 服务启动时会自动启动全局调度器

随着多人联机需求明确，需要引入“房间机制”作为后续 Steam 分发和多人游戏的基础抽象。

本设计文档的目标是：

- 将“一台服务器开的游戏”定义为一个独立的房间/存档/对局
- 支持当前开发阶段通过局域网手动加入房间
- 为未来 Steam Lobby / SteamID 集成预留兼容结构
- 尽量避免一次性大重构，采用可分阶段落地的方案

## 2. 设计原则

### 2.1 核心原则

- 一个房间就是一局游戏的完整世界状态
- 一个房间对应一个存档槽位
- 房间状态必须由本项目后端维护，不能依赖第三方平台托管核心游戏状态
- Steam 未来只承担“发现、邀请、身份辅助、连接辅助”的职责，不承担游戏世界状态职责
- 前后端、数据库、事件广播、身份体系都要具备 `room_id` 维度

### 2.2 实施原则

- 优先支持 LAN 开发模式
- 保留未来接入 Steam 的扩展点
- 在现有架构上做最小但正确的抽象
- 优先隔离世界状态，再逐步增强房间生命周期、权限和联机体验

## 3. 术语定义

### 3.1 房间 `room`

房间是一个正在运行或可恢复的游戏世界实例。

职责：

- 承载一局游戏的市场、新闻、玩家账户、订单、合约、主持人配置等完整状态
- 负责玩家加入、离开与在线关系
- 对应一个可保存和恢复的存档

### 3.2 存档 `save_slot`

存档是某个房间对应的持久化世界快照。

职责：

- 支持房主从已有世界继续开房
- 支持房间关闭后保留进度
- 为未来云存档或版本迁移提供基础

### 3.3 房主 `host`

房主是创建房间并启动世界的人。

职责：

- 创建房间
- 配置该房间的 API/LLM 参数
- 启动与关闭房间
- 管理可见性与访问权限

### 3.4 玩家 `player`

玩家是加入某个房间的人。

职责：

- 在房间范围内拥有自己的账户、订单、消息和身份
- 使用自己的显示名参与当前房间
- 可以与平台身份绑定，但不依赖平台身份作为唯一显示名

## 4. 总体架构

建议采用以下模型：

- 一个房间 = 一局游戏 = 一个存档 = 一个独立世界状态
- 一个房间运行在一个后端实例的一个逻辑分区内
- 当前 LAN 阶段由房主启动服务并开放局域网访问
- 未来 Steam 阶段，通过 Steam Lobby 做房间发现和邀请
- 真正的世界状态仍由项目后端维护

可以概括为：

```text
Steam / LAN 发现层
    ↓
房间接入层（room join / room metadata / identity binding）
    ↓
房间作用域 API 层（所有请求带 room_id）
    ↓
房间作用域数据层（每房间独立数据库/存档）
    ↓
房间作用域调度器（新闻/市场/做市/托管/规则）
```

## 5. 核心数据模型

### 5.1 Room

建议字段：

- `room_id`: 房间唯一标识
- `name`: 房间名称
- `host_player_id`: 房主游戏内 ID
- `host_platform`: 平台类型，如 `local` / `steam`
- `host_platform_user_id`: 平台用户唯一标识，未来可为 SteamID
- `visibility`: 可见性，建议支持：
  - `lan`
  - `private`
  - `steam_friends`
  - `public`
- `join_code`: 房间加入码
- `status`: 房间状态，建议支持：
  - `draft`
  - `running`
  - `closed`
- `save_slot_id`: 当前绑定的存档
- `server_version`: 服务版本
- `created_at`
- `started_at`
- `closed_at`

### 5.2 SaveSlot

建议字段：

- `save_slot_id`
- `room_id`
- `title`
- `world_seed`
- `world_version`
- `snapshot_path` 或快照元信息
- `created_at`
- `updated_at`
- `last_played_at`

### 5.3 RoomMember

建议字段：

- `room_id`
- `player_id`
- `platform`
- `platform_user_id`
- `display_name`
- `role`
  - `host`
  - `player`
  - `observer`
- `joined_at`
- `last_seen_at`
- `is_online`

## 6. 身份体系设计

## 6.1 当前问题

当前系统里大量逻辑直接使用：

- `player_id`
- `account_id = user:{player_id}`

这意味着身份默认是全局唯一，无法自然支持：

- 同名玩家存在于不同房间
- 同一玩家进入多个房间
- 平台身份与游戏内身份分离

## 6.2 推荐身份分层

建议区分三层身份：

### 平台身份

用于外部平台识别用户。

字段建议：

- `platform`: `local` / `steam`
- `platform_user_id`

说明：

- 当前 LAN 阶段可为空或使用本地生成 UUID
- 未来 Steam 阶段直接映射 SteamID

### 游戏内身份

用于当前房间中的玩家名。

字段建议：

- `player_id`
- `display_name`

说明：

- 当前阶段手动输入
- 未来可默认取 Steam 昵称，但仍建议允许房间内使用独立显示名

### 房间作用域身份

用于业务主键与所有数据库关联。

建议：

- 业务逻辑不要再使用全局 `user:{player_id}`
- 推荐使用：
  - `room:{room_id}:user:{player_id}`
  - 或单独引入 `room_member_id`

推荐优先方案：

- 仍保留 `player_id` 供前端展示
- 后端内部账户 ID 改为房间作用域账户 ID

## 7. 数据隔离设计

## 7.1 方案对比

### 方案 A：单库多表加 `room_id`

优点：

- 表面上迁移路径简单

缺点：

- 几乎所有表都要补 `room_id`
- 所有查询都必须严格带过滤
- 很容易遗漏，形成串房间数据风险
- 存档导出/备份/迁移体验差

### 方案 B：每房间一个 SQLite 数据库

建议数据库路径示例：

- `data/rooms/{room_id}/ledger.db`

优点：

- 隔离最彻底
- 一个房间就是一份完整世界数据
- 存档、备份、复制、恢复都非常自然
- 更适合当前项目“世界状态整体独立”的结构
- 更适合 LAN host 场景

缺点：

- 需要重构数据库连接获取方式
- 需要引入 room 上下文

## 7.2 推荐结论

推荐采用：**每房间一个 SQLite 数据库**。

原因：

- 当前项目本质上是“整局世界模拟”
- 世界状态天然适合整体隔离
- 后续要做房间存档时，这条路最清晰

## 8. API 作用域设计

## 8.1 总体原则

未来所有会读取或修改游戏世界状态的 API，都必须带有 `room_id`。

包括但不限于：

- 玩家引导与入局
- 账户信息
- 市场行情
- 下单与撤单
- 新闻系统
- 合约系统
- 托管系统
- 聊天系统
- WebSocket 订阅

## 8.2 room_id 的传递方式

推荐采用：

- HTTP Header：`X-Room-Id`

理由：

- 对现有 URL 改动最小
- 前端可在 API Client 层统一注入
- 不会把路径层一次性改得过大

未来如果需要更显式 REST 风格，再考虑演进为：

- `/rooms/{room_id}/players/bootstrap`
- `/rooms/{room_id}/market/summary`

## 8.3 建议的阶段性策略

第一阶段：

- 在所有关键 API 支持 `X-Room-Id`
- 未提供时可以回退到默认房间，便于过渡

第二阶段：

- 新逻辑强制要求提供 `room_id`
- 逐步淘汰默认全局房间逻辑

## 9. WebSocket 设计

当前全局频道广播方式不适合房间隔离。

## 9.1 推荐频道命名

将频道变为房间作用域，例如：

- `room:{room_id}:events`
- `room:{room_id}:presence`
- `room:{room_id}:TRADE_EXECUTED`
- `room:{room_id}:NEWS_PUBLISHED`

## 9.2 目标

- 不同房间用户不会收到彼此事件
- presence 统计按房间独立
- 后续可以扩展到每房间聊天频道和系统广播

## 10. 调度器与世界运行模型

当前项目在服务启动时直接启动全局调度器：

- 合约调度器
- 新闻调度器
- 市场时段调度器
- 做市商调度器
- 托管调度器

这意味着当前模型是“服务一启动，全局唯一世界就开始运行”。

## 10.1 推荐方向

将调度器改为“房间作用域调度器”。

每个运行中的房间应拥有独立的：

- 新闻 tick
- 市场/时间推进
- 做市商行为
- 托管行为
- 规则扫描

## 10.2 房间运行状态

建议房间状态流转：

- `draft`: 房间已创建，但尚未启动世界
- `running`: 世界正在运行，允许玩家加入和参与
- `closed`: 房间已关闭，世界停止运行，仅保留存档

## 10.3 房间启动方式

房主执行“启动房间”后：

- 初始化该房间数据库/存档
- 启动该房间对应的调度器集合
- 开放加入入口

房主执行“关闭房间”后：

- 停止该房间调度器
- 落盘存档
- 切换为 `closed`

## 11. 游戏时间与市场会话

当前 `game_time` 与 `market_session` 默认按全局配置和全局元数据工作。

## 11.1 问题

如果未来存在多个房间：

- 不同房间可能需要不同时间设置
- 不同房间必须拥有独立 epoch
- 世界推进不能共享

## 11.2 推荐方案

游戏时间配置与市场时间状态改为房间作用域：

- 每房间维护自己的 `epoch_utc`
- 每房间维护自己的时间配置
- 每房间维护自己的 market session 状态

## 12. 房间设置与 LLM 配置

## 12.1 当前需求

你已经明确：

- 房主配置对应 API
- 房主启动房间
- 其他玩家加入并共同参与该局游戏

因此，LLM/供应商/API Key 不应视作全局用户设置，而应至少支持“房间级配置”。

## 12.2 推荐分层

### 本机设置

保存在本机，仅对当前安装实例生效：

- 默认语言
- 本机 UI 偏好
- 默认网络模式
- Steam 启用开关

### 房间设置

与房间/存档绑定：

- LLM provider 配置
- API key 或密钥引用
- 任务路由
- 世界参数
- 市场参数
- 联机可见性

### 玩家设置

与玩家自身绑定：

- 语言
- 涨跌颜色
- 前端布局
- 个人显示偏好

## 12.3 安全性建议

- 房主配置的 API Key 不应以明文同步给其他玩家端
- 玩家端只调用房主提供的后端服务
- 房间 LLM 能力由房主服务端代理

## 13. 前端运行模式设计

推荐前端支持两种模式：

### Host 模式

用于房主：

- 创建房间
- 配置房间参数
- 配置 LLM/API
- 启动/关闭房间
- 可选择读取某个旧存档开房

### Client 模式

用于加入者：

- 输入服务器地址
- 输入 `room_id` 或加入码
- 输入 `player_id`
- 加入房间后进入正常游戏界面

## 13.1 为什么推荐双模式而不是两套程序

优点：

- 当前开发成本更低
- 兼容“像泰拉瑞亚一样同一程序既能开房又能进房”
- 未来仍可以拆 Dedicated Server，而不会破坏核心模型

## 14. LAN 阶段设计

当前开发阶段优先支持局域网加入。

## 14.1 建议流程

### 房主侧

- 启动应用
- 选择“创建房间”
- 设置房间名称、加入码、房主 `player_id`
- 配置本局 LLM/API
- 启动房间
- 将局域网地址和加入码告诉其他玩家

### 玩家侧

- 启动应用
- 选择“加入房间”
- 输入服务器地址
- 输入 `room_id` 或加入码
- 输入自己的 `player_id`
- 完成房间加入和引导流程

## 14.2 LAN 阶段暂不要求

- 自动发现局域网主机
- NAT 穿透
- 复杂权限系统
- 跨房间迁移

## 15. Steam 阶段设计

未来通过 Steam 分发后，可接入 Steam 功能，但不要改变房间核心结构。

## 15.1 Steam 负责的能力

- 房间发现
- 好友邀请
- 好友可见性
- 成员列表同步
- 平台身份识别（SteamID）
- 可能的连接辅助

## 15.2 Steam 不负责的能力

- 游戏世界状态存储
- 房间核心数据模型
- 市场/账户/新闻/合约逻辑
- 房间存档管理

## 15.3 Steam 集成建议

Steam lobby metadata 建议至少包含：

- `room_id`
- `room_name`
- `server_addr`
- `join_code`（如需要）
- `build_version`
- `visibility`

客户端通过 Steam 加入 lobby 后，仍应读取这些元数据，再去连接你的后端服务。

## 16. 存档生命周期设计

建议明确“房间”和“存档”的关系。

### 推荐关系

- 一个房间绑定一个当前激活存档
- 同一个存档可在未来再次被房主加载并重新开房
- 房间关闭后保留存档，不保留运行态调度器

### 推荐操作

- 新建存档并开房
- 从已有存档开房
- 关闭房间并保存
- 只读查看历史存档元信息

## 17. 建议的迭代路线

## Phase 1：建立房间作用域基础

目标：让系统从“全局单局”变成“至少逻辑上支持多房间”。

建议内容：

- 前端 session 增加 `roomId`
- API 请求统一支持 `X-Room-Id`
- 数据库访问支持按房间选库
- WebSocket 频道按房间隔离
- 玩家账户 ID 改为房间作用域

## Phase 2：建立房间生命周期

目标：让“房主创建房间并启动世界”成为正式流程。

建议内容：

- 新增创建房间、启动房间、关闭房间接口
- 新增房间元数据存储
- 新增存档槽位和房间绑定逻辑
- 调度器改为按房间启动/关闭

## Phase 3：建立加入流程

目标：让 LAN 加入过程完整可用。

建议内容：

- 新增加入房间 UI
- 输入服务器地址 + 加入码 + `player_id`
- 房间内完成 bootstrap
- 首屏数据预热仍保留并升级为房间作用域

## Phase 4：接入 Steam

目标：将 Steam 作为社交/发现层接入。

建议内容：

- 引入 `platform = steam`
- 将 `platform_user_id` 映射为 SteamID
- Steam Lobby 与 `room_id` 双向绑定
- 支持好友邀请与通过 Steam 进房

## 18. 非目标

当前阶段不建议立即解决以下问题：

- 完整 Dedicated Server 拆分
- 云存档同步
- 跨服务器迁移房间
- 跨房间角色继承
- 复杂权限和踢人/封禁体系
- 大规模公服发现与匹配系统

## 19. 风险与注意事项

### 数据兼容风险

当前大量逻辑默认全局数据，迁移过程中必须系统性检查：

- API
- 数据层
- 调度器
- 广播层
- 前端缓存键
- 本地存储键

### 身份兼容风险

若继续使用全局 `player_id` 作为主键，后续房间化会越来越难迁移。

### 设置泄露风险

房主的 API Key 和 LLM 配置必须视作服务端能力，不能简单下发给客户端。

### 事件串房风险

在房间广播改造完成前，不应允许多个房间同时运行于同一逻辑空间。

## 20. 推荐结论

推荐采用以下正式方案：

- 房间 = 一局游戏 = 一个存档 = 一个世界状态
- 核心状态由项目后端维护
- 当前先支持 LAN 手动加入
- 未来通过 Steam 做房间发现、邀请与 SteamID 绑定
- 采用“每房间一个 SQLite 数据库”实现世界隔离
- API / WS / 调度器 / 身份体系都引入 `room_id` 作用域
- 程序运行形态优先采用同一客户端支持 Host / Client 双模式

这是兼顾当前开发成本、未来 Steam 分发、以及多人房间长期可维护性的最稳妥路线。

## 21. 下一步实施建议

推荐下一步按以下顺序落地：

1. 引入 `room_id` 并打通前后端 session 传递
2. 将数据库改造为按房间选库
3. 将 WebSocket 改造为按房间广播
4. 将账户 ID 改造为房间作用域
5. 新增创建/加入房间的最小 UI 与 API
6. 将房主配置的 LLM 设置提升为房间级配置
7. 最后再接 Steam Lobby 和 SteamID
